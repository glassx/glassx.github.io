MVC、MVP、MVVM 的理解

MVC

典型的MVC：

  • 由view接受指令，将指令传递到Controller
  • Controller 完成逻辑后，要求Model 改变状态
  • Model 将新数据发送到View，用户得到反馈

一般的流程： view 触发事件 –> controller处理了业务，然后触发数据跟新–> model本身不知道谁更新了数据–>model 带着数据回到view –> view更新

MVP

MVP典型的交互如下图所示：

MVP一般遵循如下规则：

  • View 与 Model 不直接发生联系
  • View 非常薄，不做任何业务逻辑
  • Presenter 非常厚，所有的逻辑操作都部署在里面
  • 可以通过 View 实现接口（这样方便将view 和 presenter 分开单独开发），或者 干脆 Presenter 持有 View的引用，这样，方便 Presenter 调用方法更新 View
  • 当然，Presenter 可以将数据存储到 Model 或者从 Model 中取出数据

MVVM

MVVM 典型的交互如下图所示：

它基本上与MVP 完全一致，唯一的区别是：它采用的是双向绑定（注意，是绑定，而不是接口或者持有引用）：

  • 当View 变动时，自动反映在ViewModel上，反之亦然。
  • 数据和业务逻辑处于一个独立的ViewModel 中，ViewModel 只需要关注业务和数据，无需与ui打交道。 由数据自动去驱动UI自动更新，同时，UI改变又自动反馈到数据。
  • 由于ViewModel 与ui没有任何关联，故ui的所有调整都不需要改动viewmodel。
  • 整个过程，数据是主导因素。

总结特性(自己胡诌的)

  • mvc 与 mvp 都是事件驱动，而 mvvm 是数据驱动
  • mvc 中，Controller 控制view的切换，处理相关控制逻辑，model 控制view的刷新。而mvp中，view 只展示，presenter 处理所有的逻辑，model 只关注数据，view更新通过Presenter 持有view的引用，或者view 实现相关接口实现。
